Critical — will cause silent bugs or crashes

DONE: No rollback on failed reload.

	If slot_load_dll succeeds but call_reload returns false, m->status is set to MODULE_STATUS_ERROR 
	and the module has no code. The old shadow DLL was already unloaded. The module is now a dead slot.
	Any other module that calls through a cached API pointer into it will crash. You need to retain the
	previous shadow DLL until the new one successfully passes call_reload, and restore it on failure. 
	The minimum change is keeping the old dll handle and shadow_count alive until you're certain the
	new one works, then releasing them.

	Cascade on_reload to dependents (this will silently corrupt pointer caches as soon as you have inter-module dependencies in dynamic builds).
	and shadow file cleanup at startup (this will cause confusing load failures in long development sessions). 

DONE: Dependents of a reloaded module get stale API pointers.

	mod_reload("example") calls on_reload() on example itself, which re-caches its own sibling 
	API pointers via MOD_FETCH_API. But any module that depends on example — 
	say game.dll caches g_example_api_ptr — now holds a pointer into the OLD shadow DLL that was 
	just unloaded. Your on_reload() contract only covers the module being reloaded, not its consumers.
	This requires either automatically cascading on_reload() to all dependents in reverse-dependency 
	order after a reload, or documenting that mod_check_reloads() must trigger on_reload() on any
	module whose dependency list includes the just-reloaded module. 
	The dependency graph is already built — g_init_order has the information.

	The Flaw: If Module A depends on Module B, and Module B hot-reloads, Module B gets a new memory address for its functions. Module A is still holding the old g_B_api_ptr, which now points to unloaded memory.

	The Fix: When a module reloads, the system must either:
	Force all modules that depend on it to also call their on_reload() to refresh their MOD_FETCH_API pointers.
	Add an explicit on_dependency_reloaded() callback to the lifecycle.

	(Most robust) Change the internal storage so that g_B_api_ptr points to a stable double-pointer managed by the module system itself, so consumers automatically see the new function addresses without doing anything.

DONE: Shadow files from crashed sessions accumulate forever.

	g_shadow_counter resets to 0 on every launch. shadow_delete only knows to delete the 
	specific counter it created. Leftover .tmp_N.dll files from a previous crash are 
	never cleaned up. After many crash-heavy development sessions the build directory
	fills with stale shadow copies, some of which might accidentally shadow-match a new 
	session's counter. mod_system_init should glob and delete all *.tmp_*.dll files in
	the root directory before anything loads.

Important — correctness gaps that compound as the engine grows

TODO: Reload does not respect dependency order for batched changes.

	When both render.dll and game.dll change simultaneously (a shared header edit),
	they both enter g_pending. They expire debounce independently and reload in arrival order, 
	not dependency order. If game depends on render and render reloads first, game's cached 
	render_api pointer becomes stale mid-session. The pending expiry logic in mod_check_reloads 
	should collect all expired entries, sort them by the existing g_init_order dependency ordering, 
	then reload in that sequence rather than popping them individually.

TODO: State layout changes are invisible to the system.
	
	Your state_ensure handles size growth correctly. What it cannot detect is a structural change: 
	a developer moves a field, changes its type, or reorders members. The preserved bytes are reinterpreted 
	against the new layout silently. A reload that should reset state appears to succeed but runs with
	corrupted data. The cheapest mitigation is a state_version field that developers manually bump when 
	layout changes, checked in on_reload to decide whether to zero-init or preserve. 
	A RELOAD_RESET_STATE flag in mod_api_t lets a module opt into a clean state on each reload 
	during development without needing the developer to track this manually.

TODO: No API struct version compatibility check.

	mod_api_t has a version field that is recorded but never validated. If a consumer compiled 
	against version 2 of example_api_t and the reloaded DLL exports version 3 with a different 
	function table layout, every call through the cached struct reads garbage. 
	The version recorded in mod_info_t should be compared against whatever version consumers 
	declared they were built against. Since you control the build, the minimum viable check 
	is asserting the version has not decreased on reload.

TODO: get_mod_api returning a valid struct doesn't mean the API is fully implemented.

	slot_load_dll validates that func_api is non-NULL, but nothing validates that individual function
	pointers inside the API struct are non-NULL. A module under development might legitimately leave 
	some entries NULL. Callers that don't check will crash. Either enforce all entries are non-NULL 
	at load time, or document the convention and add a debug-mode validation pass that logs which 
	entries are NULL immediately after a successful load.

Design gaps to address before the system scales

TODO: No cooperative reload protocol.

	Currently a module gets exit() called and must immediately comply. 
	If a module is mid-frame with GPU commands in flight, audio buffers submitted, 
	or async jobs queued into rt_jobs, it has no way to say "not yet." A can_reload 
	query — a callback in mod_api_t that returns false if the module is mid-operation
	— lets the reload system defer rather than forcing it. 
	This becomes important once render and audio modules exist.

TODO: The module system is not thread-safe and that is undocumented.

	g_modules, g_pending, and all slot operations are unprotected. 
	Once rt_jobs is live and running module ticks on worker threads, any reload 
	triggered from the main thread races with tick. The current design is implicitly
	single-threaded. This is a reasonable constraint but it needs to be explicit so 
	future contributors don't accidentally add threaded ticks without addressing it. 
	The minimum is a comment in mod.h; the proper fix when jobs are introduced is a 
	read/write lock around the module registry with reloads taking the write side.

TODO: No startup validation of the full dependency graph.

	build_init_order catches cycles and missing dependencies, which is good. 
	What it doesn't catch is a module declaring a dependency that exists but is not 
	yet loaded when mod_init_all is called. If the host registers modules in the wrong 
	order before calling mod_init_all, the dependency is listed in the graph but the 
	dependent module might have been registered before its dependency. The current slot 
	scan handles this correctly only because it scans all slots. But nothing validates 
	that every name in deps[] actually resolves to a loaded slot before the sort begins 
	— the error only surfaces mid-sort when slot_find returns -1. Explicit pre-validation
	of all declared dependency names before the sort would give a cleaner error 
	message and catch typos in dep names early.

TODO: on_reload is required for hot-reload but only discovered at reload time.

	A module is registered, loaded, initialized, and runs fine. The first time 
	it changes on disk, call_reload discovers on_reload is NULL and fails. From the 
	developer's perspective the module has been "working" for hours and suddenly stops reloading. 
	This should be caught at load time for dynamic modules, not at first reload. 
	slot_load_dll or mod_dynamic_load should log a clear warning when mod_api->reload 
	is NULL: "module loaded but will not support hot-reload."

Forward-looking gaps worth designing for early

TODO: No module event system.

	Right now the only way a module knows another module reloaded is if its own
	on_reload is called. As module count grows you'll want game.dll to respond 
	to render.dll reloading without being an explicit dependent. A lightweight 
	publish/subscribe within the module system — mod_subscribe(name, callback) — 
	lets modules express interest in peer lifecycle events without hard-wiring dependencies. 
	This also enables an editor overlay to show which modules reloaded and when.

TODO: No persistent identity for module instances.

	If you ever want multiple instances of the same module (unlikely but not impossible for sandboxing),
	or if you want editor tooling to track a module across reloads (same module, version 1 vs version 2),
	the current name-keyed design has no concept of instance identity separate from name.
	Worth deciding explicitly that a module name is its identity and that the system is 
	intentionally singleton-per-name, so this is a stated constraint rather than an oversight.

TODO: State binary compatibility across compiler upgrades.

	State blocks are preserved in memory across reloads within a session, not across launches.
	This is fine. But if you ever persist state to disk between launches (save/restore for rapid iteration),
	you need a serialization layer, not raw memcpy. Worth deciding that in-memory preservation 
	is the only guarantee and documenting it.

The most urgent three to address now

FIRST: If you're hardening this before building the rest of the engine on top of it, the order of priority is:

	Cascade on_reload to dependents (this will silently corrupt pointer caches as soon as you have inter-module dependencies in dynamic builds).
	and shadow file cleanup at startup (this will cause confusing load failures in long development sessions). 
	
	The function-pointer-in-state rule is equally critical but is enforced by convention rather than code
	— document it clearly in mod_export.h where state structs are defined.


TODO: The topological sort handles arbitrary dependencies well, but it doesn't enforce architectural boundaries.

	The Fix: To maintain strict separation between core engine modules and game-specific project modules, 
	the dependency resolver should be aware of module categories. A dedicated "projects" module
	should never be a dependency of an "engine" module. Adding a simple category flag or path 
	prefix constraint to the topological sort can prevent spaghetti dependencies from forming
	as the team size grows.
